# Справочник по API Оркестратора

Этот документ описывает все HTTP-эндпоинты, предоставляемые сервером Оркестратора. API разделен на три логические группы: Публичные, Клиентские и Внутренние (для Воркеров).

## Аутентификация

-   **Клиент -> Оркестратор:** Все запросы к клиентским эндпоинтам (префикс по умолчанию `/api/v1/*`) должны содержать заголовок `X-Client-Token` с токеном клиента. Часть пути `/api` настраивается через `CLIENT_API_PREFIX`.
-   **Воркер -> Оркестратор:** Все запросы к эндпоинтам `/_worker/*` должны содержать заголовок `X-Worker-Token` с валидным токеном воркера.

---

## 1. Публичные эндпоинты (`/_public`)

Эти эндпоинты не требуют аутентификации и всегда используют фиксированный префикс `/_public`.

### Проверка статуса сервиса
-   **Эндпоинт:** `GET /_public/status`
-   **Описание:** Подтверждает работоспособность сервиса.
-   **Ответ (`200 OK`):** `{"status": "ok"}`

### Вебхук "Aprobación Humana"
-   **Эндпоинт:** `POST /_public/webhooks/approval/{job_id}`
-   **Описание:** Ручное подтверждение шага пайплайна.
-   **Ответ (`200 OK`):** `{"status": "approval_received"}`

---

## 2. Клиентские эндпоинты (`/{CLIENT_API_PREFIX}/v1`)

Требуют заголовок `X-Client-Token`.
Базовый путь для этих эндпоинтов настраивается через переменную окружения `CLIENT_API_PREFIX` (по умолчанию `api`). Если установить пустую строку, эндпоинты будут доступны от корня (например, `/v1/...`).

### Создание нового задания
-   **Эндпоинт:** `POST /{CLIENT_API_PREFIX}/v1/{blueprint_api_endpoint}`
-   **Описание:** Запуск нового экземпляра (Job) блупринта.
-   **Тело запроса:**
    ```json
    {
      "initial_data": { ... },
      "webhook_url": "https://callback.url/webhook",
      "dispatch_timeout": 60,
      "result_timeout": 300
    }
    ```
-   **Ответ (`202 Accepted`):** `{"status": "accepted", "job_id": "..."}`

### Получение статуса задания
-   **Эндпоинт:** `GET /api/v1/jobs/{job_id}`
-   **Описание:** Возвращает текущее состояние задания. Детализация ответа зависит от настройки сервера `DETAILED_API_RESPONSES`.

#### Статусы заданий (Job Statuses)
Система использует следующие статусы для управления жизненным циклом заданий:

**Промежуточные состояния:**
*   `pending`: Задание создано и ожидает постановки в очередь выполнения.
*   `running`: Задание активно обрабатывается.
*   `waiting_for_worker`: Ожидание освобождения или подключения подходящего воркера.
*   `waiting_for_human`: Ожидание действия или подтверждения от человека (Human-in-the-loop).
*   `waiting_for_parallel`: Ожидание завершения всех подзадач в параллельной ветке.

**Финальные состояния (Terminal States):**
*   `finished`: Задание успешно завершено. Доступен финальный результат.
*   `failed`: Ошибка в логике или на стороне воркера. Доступна информация об ошибке.
*   `cancelled`: Задание отменено пользователем или системой.
*   `error`: Критическая системная ошибка при выполнении.
*   `quarantined`: Задание помещено в карантин для ручного разбора (например, при нарушении контракта данных).

-   **Параметры:** `?fields=status,result` — фильтрация возвращаемых полей (в компактном режиме ограничена разрешенным списком).

**Примечание по безопасности:** Поле `result` заполняется и становится доступным только тогда, когда задача достигает финального состояния (`finished` или `failed`). Для задач в промежуточных состояниях (`running`, `waiting`) результат скрыт для защиты промежуточных данных.

**Изоляция данных:** Доступ к задаче строго ограничен клиентом, который её создал. Вы должны предоставить тот же `X-Client-Token`, который использовался при запуске. Любая попытка доступа к ID задачи, принадлежащей другому клиенту, вернет `404 Not Found` (в целях безопасности, чтобы предотвратить перебор ID).

**Пример компактного ответа (по умолчанию):**
```json
{
  "id": "123e4567-e89b-12d3-a456-426614174000",
  "status": "finished",
  "result": { "output": "success" },
  "blueprint_name": "my_flow"
}
```

**Пример детального ответа (при DETAILED_API_RESPONSES=true):**
```json
{
  "id": "...",
  "status": "finished",
  "result": { ... },
  "blueprint_name": "...",
  "state_history": { "step1": { ... } },
  "initial_data": { ... },
  "transition_count": 5
}
```

### Работа с S3
-   `GET /api/v1/jobs/{job_id}/files/upload` — получение ссылки для загрузки.
-   `PUT /api/v1/jobs/{job_id}/files/content/{filename}` — прямой стриминг файла.
-   `GET /api/v1/jobs/{job_id}/files/download/{filename}` — стабильная ссылка на скачивание.

### Управление и диагностика
-   `POST /api/v1/jobs/{job_id}/cancel` — отмена активной задачи.
-   `GET /api/v1/jobs/{job_id}/history` — полная история событий задания. В компактном режиме снимки состояния (`context_snapshot`) внутри событий фильтруются.
-   `GET /api/v1/blueprints/{blueprint_name}/graph` — структура блупринта в формате DOT.
-   `GET /api/v1/workers` — список всех активных воркеров.
-   `GET /api/v1/dashboard` — агрегированная статистика системы.
-   `GET /api/v1/workers/catalog` — каталог навыков (кэш 10 сек).
-   `GET /api/v1/jobs` — список всех задач. В компактном режиме снимки состояния каждой задачи фильтруются.

## 3. Внутренние эндпоинты для Воркеров (`/_worker`)

Требуют заголовок `X-Worker-Token`.

### Регистрация и Heartbeat
-   **POST /_worker/workers/register** — регистрация воркера и его возможностей.
-   **PATCH /_worker/workers/{worker_id}** — подтверждение активности (Heartbeat). Возвращает `next_heartbeat_jitter_ms`.

### Задачи и результаты
-   **GET /_worker/workers/{worker_id}/tasks/next** — получение новой задачи (Long-Polling + Work Stealing).
-   **POST /_worker/tasks/result** — отправка результата выполнения задачи.
-   **POST /_worker/events** — отправка промежуточного события (прогресс, логи).

### WebSocket
-   **GET /_worker/ws/{worker_id}** — канал для команд реального времени (например, `cancel_task`).
