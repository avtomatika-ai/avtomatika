# Протокол распределенной трассировки (Оркестратор ↔ Воркер)

Этот документ описывает логику сквозной передачи данных телеметрии между Оркестратором и Воркерами с использованием протокола RXON и стандартов OpenTelemetry.

---

## 1. Последовательность работы

Система гарантирует, что единая «нить трассировки» сопровождает задачу от первоначального вызова API до фактического выполнения на удаленном воркере и обратно.

### Шаг 1: Диспетчеризация задачи (Оркестратор)

Когда `JobExecutor` решает отправить задачу воркеру:

1.  Создается дочерний спан для операции диспетчеризации.
2.  Используется `TextMapPropagator` из OpenTelemetry для «инъекции» контекста текущего спана в словарь.
3.  Этот словарь сохраняется в поле `tracing_context` полезной нагрузки задачи.
4.  **Опциональность**: Если телеметрия в Оркестраторе отключена, поле `tracing_context` будет пустым или отсутствовать.

### Шаг 2: Получение задачи (Воркер)

Когда Воркер получает задачу через ответ на запрос `poll`:

1.  Он проверяет наличие ключа `tracing_context` в данных задачи.
2.  **Логика**:
    - **Если контекст есть И OTel включен на Воркере**: Воркер «извлекает» (extract) контекст и запускает свой собственный спан как дочерний по отношению к спану Оркестратора.
    - **Если контекста нет ИЛИ OTel выключен на Воркере**: Воркер выполняет задачу в обычном режиме без каких-либо затрат ресурсов на телеметрию.

### Шаг 3: Отправка результата (Воркер)

После завершения выполнения:

1.  **Если OTel включен на Воркере**: Воркер внедряет (inject) контекст своего текущего спана в поле `tracing_context` сообщения `result`.
2.  **Если OTel выключен**: Сообщение `result` отправляется без метаданных трассировки.

### Шаг 4: Обработка результата (Оркестратор)

Когда Оркестратор получает сообщение `result`:

1.  Он проверяет наличие `tracing_context` в присланном результате.
2.  **Логика**:
    - **Если контекст есть**: Оркестратор создает связь (Link) между полученным контекстом и своим спаном «Обработки результата». Это создает визуальный «прыжок» в Jaeger между двумя системами.
    - **Если контекста нет**: Оркестратор продолжает внутреннюю трассировку. Никаких ошибок или предупреждений не генерируется.

## 2. Маппинг в протоколе RXON

Поле `tracing_context` является стандартным полем в корне объектов задачи (task) и результата (result).

**Пример задачи (Оркестратор → Воркер):**

```json
{
  "task_id": "...",
  "type": "inference",
  "tracing_context": {
    "traceparent": "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01"
  },
  "params": { ... }
}
```

**Пример результата (Воркер → Оркестратор):**

```json
{
  "job_id": "...",
  "status": "success",
  "tracing_context": {
    "traceparent": "00-4bf92f3577b34da6a3ce929d0e0e4736-af34902b70f067aa-01"
  },
  "data": { ... }
}
```

## 3. Отказоустойчивость и стабильность

Система спроектирована по принципу **Zero-Friction** (нулевое трение):

1.  **Работа без зависимостей**: Оркестратор использует No-Op заглушки. Если воркер прислал контекст трассировки, но в оркестраторе не установлен SDK OpenTelemetry, контекст просто игнорируется на уровне логики без падений и затрат производительности.
2.  **Версионность протокола**: Поле `tracing_context` игнорируется старыми воркерами, которые его не поддерживают (стандартное поведение JSON).
3.  **Безопасность**: Данные трассировки считаются метаданными и не используются для логики авторизации. Если воркер пришлет поддельный контекст, это повлияет только на визуализацию в Jaeger, но не на безопасность системы.

---

## 4. Чек-лист для реализации в Worker SDK

Для полной поддержки обсервабилити SDK воркера должен:

1.  [ ] Проверять наличие `tracing_context` в полученных задачах.
2.  [ ] Если OTel доступен, использовать `extract(tracing_context)` для установки родительского спана.
3.  [ ] Создавать спан с именем `Worker:Execute:{task_type}`.
4.  [ ] Перед отправкой результата вызвать `inject(result_payload['tracing_context'])`.
5.  [ ] Гарантировать, что все вызовы OTel обернуты в `try/except` для предотвращения падения воркера при сбоях библиотеки телеметрии.
